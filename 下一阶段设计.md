## 0. 评审决策

本版文档已吸收 2026-07-16 的设计评审。处理原则如下。

### 0.1 直接采纳并写入实施规格

- 拆分 0.2.0 与 0.2.1，消除范围前后不一致。
- 明确扫描模型必须提供相对路径、文件大小、documentId 与 MIME。
- 写死章节匹配评分、歧义阈值和弱匹配确认规则。
- 明确重扫事务顺序，规避 `(bookId, index)` 唯一索引中间态冲突。
- 正在播放时应用重扫结果必须刷新 Media3 播放列表，不依赖旧列表自然切换。
- 明确 `EndOfChapter > autoPlayNext > 默认连播`。
- 自动完成状态只由 PlaybackService 串行写入；UI 只负责手动标记。
- 明确 rescan 与 reauth 的入口和权限边界。
- 补充 SessionCommand、PlayerUiState、迁移与契约测试规格。

### 0.2 调整后采纳

- 不为历史章节生成看似可靠的伪 `stableKey`。v3→v4 后旧章节的 `stableKey` 保持 null，首次成功重扫后再写入真实指纹。
- `stableKey` 使用 SHA-256，而不是 SHA-1；它只是扫描指纹，真正长期稳定的应用内身份仍是保留下来的 `Chapter.id`。
- 完成状态字段随 v4 建表，但自动完成、批量操作和完成筛选在 0.2.1 开放。
- 书签不使用数据库“时间桶”唯一约束，只在应用层做短时间重复点击合并。

### 0.3 本阶段忽略或延期

- `sortTitle`：没有明确的忽略前缀/序号规则，0.2 不增加。
- 章节标题全文搜索：0.2.0 仅搜索书名和作者，章节搜索延后。
- 置顶与网格/列表切换：放到 0.2.1，不阻塞 0.2.0。
- “当前文件播完后再应用重扫”：对 URI 已变化的播放列表不可靠，不采用。
- 书签数据库唯一索引：可能误伤用户刻意添加的相邻书签，不采用。

## 1. 阶段目标

0.1 已完成本地听书的核心闭环：

```text
选择目录 → 扫描成书 → 章节连播 → 后台播放
→ 断点续听 → 倍速 → 快进快退 → 睡眠定时
```

0.2 的目标不是继续扩展播放器控件，而是让听匣适合长期管理和收听大量本地书籍：

1. 原目录内容变化后可以安全更新书籍。
2. 用户可以记录重要时间点。
3. 书籍多起来后仍能快速查找和整理。
4. 睡前听书体验更完整。
5. 不同书籍可以保留各自的播放习惯。
6. 能明确看到章节和整本书的完成状态。

### 成功标准

- 更新目录不会产生重复书籍或无故丢失当前进度。
- 书签可以在重启后恢复并准确跳转。
- 100 本书的书架仍可快速搜索和排序。
- 睡眠定时支持“本章结束”及渐弱暂停。
- 切换书籍时自动恢复该书上次使用的倍速。
- 章节完成状态与当前播放位置保持一致，并允许用户手动修正。

## 3. 核心设计原则

### 3.1 源目录仍是事实来源

- 音频文件继续使用 SAF URI，不复制到应用目录。
- 重新扫描只更新 Room 索引，不修改或删除用户源文件。
- 用户从书架移除书籍时，仍不删除源目录内容。

### 3.2 播放进度与实际收听状态分离

以下概念不能混为一谈：

- 当前播放位置：下次从哪里继续。
- 章节完成状态：用户是否认为该章已经听完。
- 实际收听时长：未来统计功能使用，本阶段暂不实现。

用户手动跳到后续章节时，当前位置可以前进，但不应自动把所有跳过章节标记为已完整听完。

### 3.3 更新目录时优先保留用户数据

文件匹配必须优先保留：

- 当前章节与位置。
- 章节完成状态。
- 书签。
- 自定义章节标题。
- 每本书播放设置。

无法可靠匹配时不得静默映射，应展示变更摘要并给出提示。

### 3.4 Service 继续作为播放状态权威

- `PlaybackService` 是播放进度的唯一持久化写入方。
- UI 不直接持有 ExoPlayer。
- 睡眠定时由 Service 执行，避免 Activity 退后台后失效。

## 4. 数据模型设计

建议数据库从当前 v3 升级到 v4。

### 4.1 Book 新增字段

| 字段 | 类型 | 默认值 | 说明 |
|---|---|---:|---|
| `playbackSpeed` | Float? | null | 本书独立倍速；null 使用全局设置 |
| `autoPlayNext` | Boolean | true | 是否自动连播下一章 |
| `lastScannedAt` | Long | 0 | 最近成功扫描时间 |

`isPinned` 在 0.2.1 需要时再通过下一次迁移增加，避免为未冻结功能提前扩表。

### 4.2 Chapter 新增字段

| 字段 | 类型 | 默认值 | 说明 |
|---|---|---:|---|
| `relativePath` | String | fileName | 相对书籍根目录路径 |
| `fileSize` | Long | 0 | 文件大小，用于重新扫描匹配 |
| `documentId` | String? | null | SAF provider 文档标识 |
| `mimeType` | String? | null | 文件 MIME |
| `stableKey` | String? | null | 扫描指纹；旧数据首次重扫后生成 |
| `customTitle` | String? | null | 用户自定义标题 |
| `completionState` | Int | 0 | 0 未听、1 收听中、2 已完成 |
| `completedAt` | Long? | null | 完成时间 |

0.2.0 展示标题规则：

```text
customTitle ?: title
```

当前扫描器尚未读取独立 `metadataTitle`，因此本阶段不引入第三套标题来源。

### 4.3 BookmarkEntity

```kotlin
@Entity(
    tableName = "bookmarks",
    foreignKeys = [
        ForeignKey(
            entity = BookEntity::class,
            parentColumns = ["id"],
            childColumns = ["bookId"],
            onDelete = ForeignKey.CASCADE,
        ),
        ForeignKey(
            entity = ChapterEntity::class,
            parentColumns = ["id"],
            childColumns = ["chapterId"],
            onDelete = ForeignKey.CASCADE,
        ),
    ],
    indices = [Index("bookId"), Index("chapterId")],
)
data class BookmarkEntity(
    @PrimaryKey(autoGenerate = true) val id: Long = 0,
    val bookId: Long,
    val chapterId: Long,
    val positionMs: Long,
    val note: String? = null,
    val createdAt: Long = System.currentTimeMillis(),
)
```

同一章节允许多个书签。为避免连续点击产生重复项，只在应用层做合并：

```text
同一章节、2 秒内创建、position 相差不超过 1 秒 → 更新已有项或忽略重复点击
```

数据库不增加 position bucket 唯一索引。

### 4.4 章节稳定标识

`stableKey` 是扫描指纹，不是数据库主键。推荐生成规则：

```text
durationSec = durationMs / 1000
stableKey = sha256(normalize(relativePath) + "|" + fileSize + "|" + durationSec)
```

时长使用整秒，避免 MediaMetadataRetriever 毫秒级抖动。

匹配评分：

| 条件 | 分数 | 处理 |
|---|---:|---|
| URI 完全一致 | 100 | 强匹配 |
| 同 provider 且 documentId 一致 | 95 | 强匹配 |
| relativePath 完全一致 | 90 | 强匹配 |
| stableKey 完全一致 | 85 | 强匹配 |
| fileName + fileSize + durationSec 一致 | 80 | 强匹配 |
| fileSize + durationSec 一致，且候选在本书中唯一 | 70 | 改名/移动候选 |
| fileName + fileSize 一致，且候选唯一 | 65 | 中等匹配 |
| 仅 fileName 一致，且双方都全局唯一 | 45 | 弱匹配，需用户确认 |
| 其他 | 0 | Added 或 Ambiguous |

判定规则：

- 最高分至少 65，且比第二名高至少 15 分：自动匹配。
- 最高分 45–64：作为弱匹配显示在确认页。
- 最高分差小于 15：`Ambiguous`，不得自动迁移进度或书签。
- 匹配成功时保留旧 `Chapter.id`，并用新扫描信息更新 stableKey。

不能只使用 `fileName`，因为不同子目录可能存在同名文件。

## 5. 增量重新扫描

### 5.0 扫描模型前置升级

现有 `ScannedChapter` 必须先扩展为：

```kotlin
data class ScannedChapter(
    val title: String,
    val uri: String,
    val documentId: String?,
    val relativePath: String,
    val fileName: String,
    val fileSize: Long,
    val mimeType: String?,
    val durationMs: Long,
    val index: Int,
    val stableKey: String,
)
```

扫描器要求：

- 递归遍历目录，不再限制一层子目录。
- 始终保留相对根目录路径。
- 排序按规范化后的 relativePath 分段自然排序。
- 扫描循环支持协程取消。
- 无法获取大小时写 0，但不得把 `size=0` 当作强匹配条件。

### 5.1 用户入口

书籍详情页右上角菜单新增：

- 重新扫描目录。
- 重新授权目录。
- 编辑书籍信息（后续小版本）。
- 从书架移除。

### 5.2 扫描流程

```text
读取当前 Book/Chapter
→ 校验 SAF 权限
→ 扫描目录生成 ScannedChapter
→ 计算新旧章节匹配关系
→ 生成变更预览
→ 用户确认
→ Room 事务更新（预览阶段绝不落库）
→ 重建章节偏移索引
→ 若当前正在播放该书，刷新播放列表并恢复位置
```

### 5.3 变更类型

```kotlin
sealed interface ChapterChange {
    data class Unchanged(...) : ChapterChange
    data class Renamed(...) : ChapterChange
    data class Added(...) : ChapterChange
    data class Removed(...) : ChapterChange
    data class Ambiguous(...) : ChapterChange
}
```

### 5.4 确认页

扫描完成后展示：

- 新增 N 章。
- 删除 N 章。
- 重命名/移动 N 章。
- 无法确认匹配 N 章。

默认行为：

- 新章节加入对应自然排序位置。
- 已移除章节从数据库删除。
- 已匹配章节保留原 ID，避免书签外键失效。
- 当前章节被删除时，选择最接近的下一章；没有下一章则选择上一章。
- 无歧义变更可一键确认。

弱匹配或 Ambiguous 项必须由用户确认对应关系；用户退出确认页时，数据库、书签和当前播放列表保持原样。

### 5.5 数据库事务顺序

`chapters` 存在 `(bookId, index)` 唯一索引。为避免章节交换顺序时撞中间态，确认后的事务必须采用以下顺序：

```text
1. 读取并再次校验 Book/Chapter 版本，防止预览期间数据变化
2. 将所有保留章节的 index 临时更新为 -(oldIndex + 1)
3. 更新 matched 章节，但暂不写最终 index
4. 删除 Removed 章节，并统计将级联删除的书签
5. 插入 Added 章节，使用最终非负 index
6. 给 matched 章节写入最终非负 index
7. 修正 currentChapterId/currentPositionMs/listenedDurationMs
8. 更新 Book.totalDurationMs/lastScannedAt
9. 提交事务
```

要求：

- 所有历史 index 必须是非负数，临时负数只存在于事务内部。
- 更新 matched 行时保留原 Chapter.id。
- 事务失败必须全部回滚。
- `offsetIndexCache` 只能在事务成功后替换，不能在事务中提前更新。
- SAF 权限获取和释放不能放进 Room 事务。

### 5.6 正在播放时重新扫描

扫描可以后台进行，但“应用变更”必须明确刷新播放器列表。

应用确认时：

```text
若当前没有播放该书
  → 直接提交数据库变更

若当前正在播放该书且当前章 matched
  → 保存 playing/position/speed
  → 极短暂停
  → 提交数据库事务
  → 用新章节列表 setMediaItems
  → seek 回 matched Chapter.id + min(oldPosition, newDuration)
  → 恢复 speed 和原播放状态

若当前章 Removed
  → 暂停并提交变更
  → 优先选择新列表中的下一章，否则上一章
  → position=0
  → 提示“当前章节已移除，已切换到……”
```

不采用“继续使用旧列表直到自然切章”，因为后续 MediaItem URI 可能已经变化。

### 5.7 rescan 与 reauth 分流

```text
Book.rootUri 权限有效且用户选择原 root → rescan
Book.rootUri 权限失效 → 只能进入 reauth
rescan 收到不同 rootUri → 拒绝并引导 reauth
reauth 成功 → 先建立新目录映射，再复用 RescanPlanner 应用章节变更
```

两个入口可以共用 Matcher/Planner，但权限获取、旧 root 释放和错误提示必须由 reauth 流程负责。

## 6. 书签与备注

### 6.1 播放器交互

全屏播放器新增书签按钮：

- 单击：立即保存当前时间点，并显示 Snackbar。
- 保存后 Snackbar 提供“添加备注”。
- 长按或二次入口：直接打开备注编辑框。

### 6.2 书签列表

书籍详情页增加“书签”区块或 Tab：

```text
第 12 章 · 18:35
这里开始解释核心概念
2026-07-16 22:10
```

操作：

- 点击：开始播放对应章节并跳转到时间点。
- 编辑备注。
- 删除。

### 6.3 播放跳转

`PlayerController.playBook()` 增加明确的 `positionMs` 调用路径；只有书签章节与当前书一致时才能直接 seek，否则应重建该书播放列表后定位。

### 6.4 边界条件

- 章节重新扫描匹配成功时书签不变。
- 章节确实被删除时，其书签随外键级联删除；确认页必须提示将删除多少书签。
- position 超过新文件时长时，截断到新时长。

书签跳转失败矩阵：

| 情况 | 行为 |
|---|---|
| Chapter 存在且 URI 可读 | 正常定位播放 |
| Chapter 存在但 root 权限失效 | 标记 needsReauth，打开重新授权入口 |
| Chapter 存在但单文件丢失 | 提示重新扫描目录，不自动删除书签 |
| 重扫弱匹配未确认 | 书签继续挂旧 Chapter，预览不落库 |
| 用户确认删除 Chapter | 确认页显示受影响书签数量，确认后级联删除 |
| position 超过新 duration | 截断并提示“书签位置已调整” |

## 7. 搜索、排序与筛选

### 7.1 搜索

书架 TopAppBar 增加搜索入口，搜索范围：

- 书名。
- 作者。

查询应防抖约 200–300ms，并由 Room 执行，不在 Compose 中对大量列表反复过滤。

章节标题搜索延后到 0.2.1；避免在 0.2.0 引入 books/chapters 大表 JOIN 和结果分组交互。

### 7.2 排序

支持：

1. 最近播放。
2. 最近导入。
3. 书名。
4. 收听进度。

0.2.0 不做置顶。0.2.1 增加置顶后，置顶书籍始终优先，置顶内部再按当前规则排序。

### 7.3 筛选

- 全部。
- 未开始。
- 收听中。
- 需要重新授权。

“已完成”筛选随章节完成状态在 0.2.1 开放。

书籍状态计算：

```text
未开始：lastPlayedAt == 0
收听中：lastPlayedAt > 0
```

0.2.1 增加完成状态后：

```text
已完成：所有有效章节 completionState == COMPLETED
收听中：已开始且未满足已完成条件
```

### 7.4 偏好持久化

DataStore 保存：

- 当前排序方式。
- 当前筛选方式。
- 网格或列表布局。

搜索词不持久化。

## 8. 睡眠定时增强

### 8.1 模式

```kotlin
sealed interface SleepTimerMode {
    data object Off
    data class AfterDuration(val durationMs: Long)
    data object EndOfChapter
}
```

优先级固定为：

```text
SleepTimerMode.EndOfChapter > Book.autoPlayNext > 播放器默认连播
```

本阶段支持：

- 15 / 30 / 45 / 60 分钟。
- 本章结束。
- 关闭。

### 8.2 本章结束

开启后：

- 当前章节自然播放结束时暂停。
- 不自动进入下一章。
- 用户手动切章后，定时目标更新为新章节结束。
- 用户关闭定时后恢复正常连播策略。

实现要求：

- Service 在定时开启时记录目标 `chapterId`。
- 目标章节收到自然结束事件时，在进入下一章前暂停。
- 不采用“已经切到下一章再 seek 回去”的实现，避免进度和完成状态竞态。
- 用户手动切章时，目标更新为新章节，不取消定时。

### 8.3 音量渐弱

倒计时模式到期前开始渐弱，渐弱时长为：

```text
fadeDuration = min(30 秒, 当前剩余倒计时)
```

渐弱期间：

- 记录原始播放器音量。
- 每秒平滑降低音量。
- 到期暂停。
- 暂停后恢复原始音量，避免下次播放仍然静音。

如果用户在渐弱期间关闭定时，应立即恢复音量。

### 8.4 延长定时

剩余 60 秒时可通过播放器界面延长 15 分钟。通知栏交互可延后，不作为 0.2.0 首发阻断项。

### 8.5 状态归属

- 定时逻辑完全位于 PlaybackService。
- UI 通过自定义 SessionCommand 设置模式。
- Service 向 Controller 暴露剩余时间和模式。
- 不要求进程死亡后恢复睡眠定时。

### 8.6 SessionCommand 与状态回传

| 方向 | 命令/字段 | 说明 |
|---|---|---|
| UI → Service | `SET_SLEEP_MODE` | `{mode, durationMs?}` |
| UI → Service | `SET_BOOK_SPEED` | `{bookId, speed?}`，null 表示恢复全局默认 |
| Service → UI state | `sleepMode` | Off / AfterDuration / EndOfChapter |
| Service → UI state | `sleepRemainingMs` | 倒计时模式剩余时间 |
| Service → UI state | `effectiveSpeed` | 当前实际倍速 |
| Service → UI state | `sleepTargetChapterId` | 本章结束模式目标章节 |

书签添加直接走 UI → BookmarkRepository，不经过 MediaSession；只有自动进度和自动完成状态由 Service 写入。

## 10. 章节完成状态

> 交付版本：0.2.1。字段随 v4 预留，0.2.0 不展示完成筛选或批量操作。

### 10.1 状态定义

```kotlin
enum class ChapterCompletionState {
    NOT_STARTED,
    IN_PROGRESS,
    COMPLETED,
}
```

Room 中使用 Int 存储，并通过转换函数映射。

### 10.2 自动状态规则

- 首次播放且 position > 0：`IN_PROGRESS`。
- 自然播放到章节结尾：`COMPLETED`。
- 用户手动跳到下一章：原章节保持 `IN_PROGRESS`，不自动完成。
- 从已完成章节重新播放：默认保持 `COMPLETED`。

0.2.1 不使用“达到 95% 自动完成”，避免拖动进度条或文件时长不准导致误判。后续如增加该能力，必须作为可关闭设置。

自动状态写入方：

| 写入方 | 允许写入 |
|---|---|
| PlaybackService 串行事件队列 | 自然结束→COMPLETED；首次有效播放→IN_PROGRESS |
| UI → UseCase | 手动标记、批量标记 |
| PlayerController | 禁止写数据库状态 |

进度和自动完成事件进入同一串行 writer。自然切章顺序固定为：

```text
写入旧章 COMPLETED
→ 写入新章 currentChapterId + position=0
```

两次更新应在 Repository 事务或单个数据库方法内完成，避免只完成其中一半。

### 10.3 手动操作

章节长按菜单：

- 标记为已听。
- 标记为未听。
- 从本章开始播放。

书籍详情菜单：

- 全部标记已听。
- 清除全部完成状态。

批量操作必须二次确认。

### 10.4 书籍完成状态

仅当所有有效章节都为 `COMPLETED` 时，书籍才算完成。

新增章节后，原本已完成的书籍会重新变为收听中，这是预期行为。

进度条与完成状态是两个不同概念：

- 进度条：当前章节在全书时间轴中的位置。
- 已完成筛选：所有章节都被明确标记为 COMPLETED。

升级到 v4 后，历史章节不推断为已完成；完成状态从升级后的真实播放事件开始积累。这是预期行为，发布说明必须明确告知。

## 13. 数据库迁移

建议 v3 → v4：

1. 为 books 增加每本书设置字段。
2. 为 chapters 增加稳定标识及完成状态字段。
3. 创建 bookmarks 表和索引。
4. 为已有章节回填 `relativePath=fileName`。
5. 旧章节 `fileSize=0`、`documentId=null`、`stableKey=null`。
6. 根据当前书籍位置初始化章节状态：
   - 当前章节之前不自动标记完成，避免错误推断。
   - 当前章节且 position > 0 标记为收听中。
   - 其他章节保持未听。

不在 Migration 中生成 `fileName|0|duration` 之类的弱指纹，避免后续被误判为强匹配。首次成功 rescan 时：

```text
使用 URI/relativePath/fileName/duration 的降级规则匹配旧章节
→ 用户确认弱匹配
→ 写入真实 relativePath/fileSize/documentId/stableKey
```

书籍详情页在首次重扫前可提示：“首次重新扫描将建立更稳定的章节匹配信息。”

必须保留：

- v1、v2、v3 历史 schema。
- `MIGRATION_1_2`、`MIGRATION_2_3`。
- 新增 `MIGRATION_3_4`。
- CI 测试 `1→4`、`2→4`、`3→4`。

## 14. 测试计划

### 14.1 JVM 单元测试

- 同 URI 章节匹配。
- 改名但大小/时长一致。
- 子目录同名文件不误匹配。
- 多个候选时输出 Ambiguous。
- 最高分差不足阈值时不得自动匹配。
- 当前章节被删除时选择正确相邻章节。
- 书签 position 截断。
- 睡眠定时状态转换。
- `EndOfChapter` 优先于 `autoPlayNext=true`。
- 自然播放完成与手动切章的状态差异。
- 进度和 completion 事件保持串行顺序。
- 搜索、排序和筛选组合。

### 14.2 Room instrumentation

- v3→v4 字段、索引和 bookmarks 表。
- v1/v2/v3 完整升级到 v4。
- 删除 Book/Chapter 后书签级联删除。
- 重新扫描事务失败时数据全部回滚。
- 两个章节交换 index 时不触发唯一索引冲突。
- Added/Removed/Matched 混合更新后 index 连续且唯一。
- 书签 position 超过新 duration 时被截断。

### 14.3 Media3/真机验证

- 后台播放时本章结束暂停。
- 30 秒渐弱并恢复音量。
- 切书恢复各自倍速。
- 锁屏、蓝牙切章后完成状态正确。
- 播放中应用重扫后 playlist、章节和 position 正确恢复。
- 当前章被删除时暂停并跳到明确的替代章节。
- 权限撤销时 rescan 被拒绝并进入重新授权流程。

### 14.4 性能基准

- 100 本书书架搜索响应小于 300ms。
- 500 章目录扫描不阻塞主线程。
- 增量扫描无变化时不批量重写全部章节。
- 500 个书签列表可以流畅滚动。
